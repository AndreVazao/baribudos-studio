# H1-H5 CONSISTENCY REVIEW V1

## Objective
Review the whole H1-H5 operational sequence as one coherent operator journey inside `feat/studio-operational-integration-v1`.

This document exists to prevent isolated improvements from becoming inconsistent when accumulated together.

---

## Why this review matters
The branch already has the roadmap for:
- H1 publish confidence
- H2 readiness credibility
- H3 story-first comfort
- H4 voice ownership / credits / variants clarity
- H5 Website control meaning

If these blocks evolve independently without consistency review, the Studio can become technically richer but operationally uneven.

---

## Global consistency questions
### Q1
Does the Studio speak one coherent operational language from story creation to Website control?

### Q2
Do the blocks reduce uncertainty, or just add more labels?

### Q3
Do the blocks guide the operator forward, or just describe status?

### Q4
Do improvements make the Studio calmer and clearer for family/team daily use?

### Q5
Did we improve clarity more than complexity?

---

## H1-H5 journey review

### H1 — Publish confidence
Check:
- state ladder is visible and understandable
- receipt feels trustworthy
- next recommended action reduces hesitation
- sent / received / materialized distinction is clear

### H2 — Readiness credibility
Check:
- blockers feel operationally real
- optional enrichments are clearly separated
- package frozen is not confused with publish-ready confidence
- Website contract meaning is understandable

### H3 — Story-first comfort
Check:
- text entry flow is obvious
- save/edit actions feel trustworthy
- next step after story actions is clear
- transition to illustrations is not premature or confusing

### H4 — Voice ownership / credits / variants
Check:
- owner vs credited name is instantly understandable
- original vs variant is visually and operationally clear
- credits logic feels trustworthy
- cast assignment is calmer and less repetitive

### H5 — Website control meaning
Check:
- health/status meaning feels operational
- reconcile/divergence feels actionable
- publication detail feels readable
- next corrective action is visible

---

## Cross-block consistency rules
### Rule 1 — same language model
Equivalent meanings should use equivalent language.
Examples:
- `Próximo passo recomendado` should stay canonical
- `Significado` should stay canonical
- positive/attention/failure language should remain consistent

### Rule 2 — same operator posture
Every block should:
- explain current state
- explain practical meaning
- explain next step where relevant

### Rule 3 — progressive calm
The Studio should feel like one guided operating system, not a set of independent technical panels.

### Rule 4 — no false confidence
Do not overstate:
- readiness
- materialization
- safety of voice usage
- health of Website reflection

### Rule 5 — next step always beats passive status
Wherever possible, prefer:
- practical next action
instead of:
- raw passive description only

## Review checklist
- [ ] H1-H5 use compatible operational language
- [ ] H1-H5 all reduce uncertainty rather than increasing noise
- [ ] H1-H5 preserve Studio as source of truth
- [ ] H1-H5 do not push complexity into Website unnecessarily
- [ ] H1-H5 feel reviewable and incremental in the current branch
- [ ] H1-H5 together improve daily usability for family/team use

---

## Red flags
Delay final merge if:
- [ ] one block uses strong operator language while another falls back to developer wording
- [ ] one block guides clearly but another block leaves the user guessing
- [ ] H1-H5 together add too much UI noise
- [ ] H1-H5 improvements are individually good but collectively inconsistent
- [ ] branch improvements exist mostly in docs but not enough in visible surfaces

---

## Exit expectation
Before final merge from `feat/studio-operational-integration-v1`, H1-H5 should feel like one single operational maturity pass, not five unrelated improvement notes.
